Chart.js v4.4.x ~200 KB
Canvas-based, highly flexible, massive community

Pros

  • Smallest full-featured bundle (~200 KB)
  • Excellent documentation & community
  • Canvas = fast rendering
  • Highly customizable
  • Zoom plugin available

Cons

  • Zoom requires separate plugin (chartjs-plugin-zoom + hammer.js)
  • Not React-native — needs react-chartjs-2 wrapper
  • No built-in brush/range selector
  • Financial chart type not included
  • Pinch zoom can conflict with page scroll on mobile

ApexCharts v3.x ~400 KB
Feature-rich, built-in brush/zoom, great defaults

Pros

  • Built-in zoom, pan, and brush with no extra plugins
  • Native datetime x-axis with smart tick formatting
  • Excellent touch & pinch-zoom support out of the box
  • Good React wrapper (react-apexcharts)
  • Candlestick / financial chart type included
  • Beautiful default tooltips

Cons

  • Heavier than Chart.js (~400 KB)
  • Some customizations require deep config knowledge
  • SVG-based (slightly slower than canvas at 700+ series)
  • Brush chart requires second chart instance (extra height)
📱 Mobile note: Pinch-to-zoom and pan work natively. Brush/selection chart lets users pick a sub-range — ideal for "last 3 months" navigation. Tooltip follows touch position well. Recommended for this use case.

Apache ECharts v5.x ~800 KB full / treeshakable
Handles massive datasets, powerful dataZoom, toolbox

Pros

  • dataZoom: slider + inside scroll, both built-in
  • Handles very large datasets efficiently (canvas)
  • Toolbox: save image, zoom reset, data view
  • Excellent mobile touch support
  • Treeshakable — only include what you use
  • Very active maintenance (Apache project)

Cons

  • Full bundle is large (~800 KB)
  • Configuration is verbose / JSON-heavy
  • React wrapper (echarts-for-react) is a thin layer, not idiomatic
  • Learning curve steeper than Chart.js
  • Tooltips less polished by default
📱 Mobile note: dataZoom slider appears at bottom — users drag to select range. "inside" dataZoom adds scroll/pinch on the chart itself. Both work simultaneously. Very performant on mobile with canvas renderer.

Lightweight Charts v4.x (TradingView) ~45 KB
Purpose-built financial time series, tiny & blazing fast

Pros

  • Smallest bundle of all options (~45 KB!)
  • Purpose-built for financial time series
  • Excellent touch & pinch-zoom on mobile
  • Very fast — handles thousands of points smoothly
  • Built-in crosshair & price tracking
  • Area series ideal for portfolio value visualization

Cons

  • No official React wrapper — must use vanilla JS in div
  • Limited chart types (financial focus only)
  • Less flexible for non-financial visualizations
  • Smaller community vs Chart.js / ECharts
  • Multi-series line charts less visually customizable
📱 Mobile note: Best-in-class mobile experience of all options — pinch to zoom, swipe to pan, tap for crosshair. Designed for TradingView which is a mobile-first financial app. Extremely responsive.

Recommendation

Best Fit for Mobile Portfolio Charts

Option A: ApexCharts

Best balance of features, mobile support, and React integration. Built-in zoom/pan/brush, good touch UX, financial chart types, and react-apexcharts is well-maintained. Closest drop-in replacement for Recharts with major UX improvements.

Option B: Lightweight Charts

Best mobile performance and smallest bundle. Purpose-built for financial data. Trade-off: no React wrapper (use useRef + useEffect). Ideal if performance and bundle size are top priorities and team is comfortable with vanilla JS integration.

Reasoning: This project renders multi-line portfolio performance data over 2 years (~730 points) and needs to work well on mobile. Recharts currently lacks zoom/pan and mobile navigation features. ApexCharts solves this with minimal migration friction (similar declarative React API). Lightweight Charts solves it even better from a performance standpoint but requires more integration work. ECharts is powerful but over-engineered for this use case. Chart.js is fine but zoom is a plugin afterthought. Plotly is eliminated by bundle size alone for a mobile-first app.
